# 주식 왕국 전쟁 서비스 - 요구사항 정의서 (MVP 1단계)

## 1. 배경 (Context)

기존 HTS/MTS는 단순 거래 기능에 머물러 있다. 같은 종목을 보유한 주주들이
"왕국"이라는 단위로 묶여 서로 경쟁하는 게이미피케이션 요소를 더해,
주식 보유를 더 재미있게 만드는 서비스를 구상한다.

이 서비스의 목표는 개인 투자자들의 결속력을 높여 대주주와 같은 효과를 얻기 위함이다.

## 2. 핵심 유저 시나리오 (MVP 기준)

> 사용자가 보유 종목/수량을 입력하면, 소속 왕국(종목 단위)이 자동 배정되고,
> 왕국 랭킹 페이지에서 시가총액 기준으로 스케일링된 왕국 전투력을 확인할 수 있다.

## 3. MVP 1단계 스코프

### 포함 (In Scope)
- 종목 = 왕국 1:1 매핑
- 사용자 보유 종목/수량 **수동 입력** (증권사 API 연동 아님)
- 시가총액/보유주식수 기반 왕국 전투력 스케일링 로직
- 왕국 랭킹 목록 조회 API + 간단한 화면
- 왕국 상세 페이지 (소속 주주 수, 총 보유 주식수, 전투력 점수)

### 제외 (Out of Scope, 2단계 이후로 이연)
- 섹터 통합전 (종목 왕국 → 섹터 단위 전쟁)
- 연합 시스템
- 실시간 대결/전투 애니메이션
- 보상 지급 로직 (포인트/코인/주식 보너스)
- 증권사 API 연동 (OAuth) — 목업 데이터로 대체
- 로그인/소셜 인증 (임시 사용자 ID 또는 간단 회원가입으로 대체)

## 4. 도메인 모델 (v2, 도메인 책임 기반 재설계)

| 엔티티 | 설명 | 주요 속성 |
|---|---|---|
| `User` | 서비스 사용자 | id, nickname, email |
| `Stock` | 종목(시장 데이터 전용) | id, ticker, companyName, currentPrice, **marketCap**(전투력 정규화용 시가총액) |
| `UserStockHolding` | 사용자별 보유 내역 | id, userId(FK), stockId(FK), quantity |
| `Kingdom` | 종목을 소유한 유저들의 모임(게임 도메인) | id, stockId(FK, 1:1), name |
| `KingdomPowerSnapshot` | 왕국의 일자별 전투력 + 순위 | id, kingdomId(FK), snapshotDate, totalHoldingQuantity, participantCount, power, **rank(int)**, **rankChange(int)** |

> **설계 변경 이력**: 처음엔 "전투력 계산"과 "순위 결정"을 서로 다른 책임으로 보고
> `KingdomPowerSnapshot`/`KingdomRanking` 두 테이블로 분리했으나, 실제로는
> 순위 계산에 별도 로직이랄 게 없고(단순 정렬) 배치 처리 순서 의존성(스냅샷 전체 계산
> 완료 후 순위 산출)만 존재해 테이블 분리의 실익이 없다고 판단, `rank`/`rankChange`
> 컬럼을 `KingdomPowerSnapshot`에 병합. (YAGNI — 분리가 필요해지면 그때 다시 분리)

> **Stock vs Kingdom 분리 이유**: `Stock`은 순수 시장 데이터(시세)만 다루고,
> `Kingdom`은 게임 도메인 개념(왕국 이름, 랭킹 등)을 다룸. 지금은 1:1이지만
> 추후 섹터 통합전 도입 시 `Kingdom`이 여러 `Stock`을 포함할 수 있도록 미리 분리.

## 5. 전투력 스케일링 로직 (확정)

- 원 지표(raw metric): `참여 유저 수 × 유저들의 보유 주식 총합 수 × 현재 주가`
- 목표: 시가총액이 큰 회사(예: 삼성전자)와 작은 회사가 참여자 수 기반으로
  비슷한 스케일에서 경쟁 가능하도록 정규화
- **정규화 방식(확정)**: `참여율 × log(참여자 수 + 1)`
  - 참여율 = `(유저 보유 주식 총합 × 현재가) / 종목 시가총액` — 회사 규모(시가총액)를 분모로 제거해, 회사 크기와 무관하게 "우리 서비스 내 참여 비중"만 반영
  - `log(참여자 수 + 1)` — 소수 유저가 몰빵해서 참여율만 튀는 것을 방지하고, 많은 사람이 결속할수록 유리하게 만듦 (배경의 "결속력" 목표와 부합)
  - 시가총액(marketCap)이 계산에 필요하므로, Stock 엔티티에 marketCap 필드 재추가 필요 (4번 도메인 모델 참고)

## 6. 기술 스택

- Backend: Spring Boot, Spring Data JPA, QueryDSL
- DB: PostgreSQL
- 배치: Spring `@Scheduled` (일 1회 전투력 재계산) — **실시간 반영 필요 여부는 추후 결정 (보류)**
- 데이터: 목업 데이터 (CSV 또는 시드 데이터로 종목/시가총액 초기 적재)
